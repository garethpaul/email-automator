# Inline MIME Body Parts

## Problem Frame

`GetMimeMessage` currently collects every `text/plain` and `text/html` MIME
part. A text attachment can therefore become the automation input, and an HTML
attachment can outrank the real plain-text message body. Reply generation must
derive content from inline message-body parts rather than attached files.

## Priorities

1. P0: Exclude MIME parts declared as attachments from reply content.
2. P1: Exclude named file parts even when their disposition is absent or
   malformed.
3. P2: Preserve the existing preference for inline HTML over inline plain text.

## Scope Boundaries

- Add a dependency-free Python 2/3-compatible body-part selector.
- Integrate the selector at the existing MIME traversal boundary.
- Add focused regressions and static integration contracts.
- Do not change Gmail requests, raw-message decoding, charset decoding, sender
  authorization, recipient rules, caching, reply construction, or delivery.
- Do not claim live Gmail, OAuth, App Engine, mailbox, cron, or delivery
  execution from offline tests.

## Implementation Units

### U1: Inline Body Selection

**Files:** `mail/body_parts.py` and `mail/list.py`

Select `text/plain` and `text/html` candidates only when they are not declared
as attachments and do not carry a filename. Keep HTML preference at the call
site so the established reply-content behavior remains otherwise unchanged.

### U2: Mutation-Sensitive Verification

**Files:** `tests/test_body_parts.py`, `scripts/check-baseline.sh`, and `Makefile`

Cover inline text, inline HTML, explicit text attachments, named parts without
a disposition, and mixed messages where an attached HTML file must not outrank
the plain-text body. Require the helper and its integration from the portable
baseline gate.

### U3: Maintained Guidance

**Files:** `AGENTS.md`, `README.md`, `SECURITY.md`, `VISION.md`, `CHANGES.md`,
and this plan

Document the local MIME selection boundary without implying control over
Gmail's API response allocation or live delivery behavior.

## Verification Plan

- Prove the focused attachment regressions fail against the current traversal.
- Run focused tests and source compilation on Python 2 and Python 3.
- Run repository-root and external-directory `make check` with explicit
  timeouts on both supported interpreters.
- Reject isolated helper, integration, regression, guidance, and completed-plan
  mutations.
- Audit exact intended paths, generated bytecode, vendored/dependency drift,
  whitespace, conflict markers, and credential-shaped additions.
