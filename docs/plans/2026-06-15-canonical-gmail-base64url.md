# Canonical Gmail Base64url

## Problem Frame

`decode_raw_message` validates the Gmail `raw` alphabet, padding shape, and
size, but Python's legacy base64 decoder accepts nonzero discarded pad bits.
For example, `AR==` decodes to the same byte as canonical `AQ==`, and `AQJ=`
decodes to the same bytes as canonical `AQI=`. Multiple textual encodings can
therefore cross a boundary documented as strict base64url validation.

RFC 4648 section 3.5 requires unused pad bits to be zero for canonical
encoding and permits decoders to reject encodings that violate that property.
The URL-safe alphabet in section 5 retains the same encoding rules.

## Priorities

1. P0: Reject padded and unpadded base64url values whose discarded pad bits
   are nonzero.
2. P1: Preserve canonical padded and unpadded Gmail values on Python 2 and
   Python 3.
3. P2: Keep the existing type, alphabet, padding, encoded-size, decoded-size,
   and MIME-parser containment boundaries unchanged.

## Scope Boundaries

- Enforce canonical equivalence in the dependency-free raw-message helper.
- Add focused one-byte and two-byte pad-bit regressions for padded and
  unpadded input.
- Extend the static baseline, maintained guidance, and completed-plan
  contracts.
- Do not change Gmail requests, OAuth, MIME parsing, message rules, caching,
  delivery, dependencies, vendored code, or App Engine configuration.
- Do not claim live Gmail, OAuth, App Engine, mailbox, cron, or delivery
  execution from offline tests.

## Implementation Units

### U1: Canonical Decoder Boundary

**File:** `mail/raw_message.py`

Inspect the final URL-safe alphabet value with the RFC 4648 one-byte and
two-byte pad-bit masks before decoding. Accept either complete trailing padding
or no trailing padding, and reject nonzero discarded bits without allocating a
second base64-sized buffer.

### U2: Mutation-Sensitive Verification

**Files:** `tests/test_raw_message.py`, `scripts/check-raw-message-boundary.py`,
and `scripts/check-baseline.sh`

Cover canonical and noncanonical one-byte and two-byte final quanta on both
supported interpreters. Require the canonicalization contract and completed
evidence from the portable baseline.

### U3: Maintained Guidance

**Files:** `AGENTS.md`, `README.md`, `SECURITY.md`, `VISION.md`, `CHANGES.md`,
and this plan.

Document that strict Gmail base64url validation rejects noncanonical pad bits
without implying control over the generated client's earlier JSON response
allocation.

## Work Completed

- Added a Python 2/3-compatible canonical pad-bit check after strict alphabet,
  padding-shape, and encoded-size validation and before decoded-body allocation.
- Preserved canonical padded and unpadded Gmail values while rejecting
  nonzero discarded pad bits in one-byte and two-byte final quanta.
- Added focused regressions, static contracts, synchronized guidance, and
  completed-plan enforcement.
